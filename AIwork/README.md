# AIwork 文档治理库

`AIwork/` 保存大模块设计、跨日实施记录和用户主动发起的 Code Review 档案。它用于追踪“为什么改、如何决定、如何验证”，不替代 `docs/` 中当前有效的产品与架构说明。

## 时间和目录

- 日期按 `Asia/Hong_Kong` 计算。
- 每个工作日使用 `AIwork/YYYY-MM-DD/`。
- 没有需要归档的工作时不创建空日期目录。
- `_templates/` 保存模板，不属于日期档案。

## 文件类型

| 类型 | 命名 | 用途 |
|---|---|---|
| 大模块主文档 | `MOD-YYYYMMDD-NNN-<ascii-slug>.md` | 唯一设计事实、批准状态、验收和提交记录 |
| 跨日工作记录 | `MOD-YYYYMMDD-NNN-WORKLOG.md` | 当天进度、偏差、验证和下一步 |
| 每日 Code Review | `CODE_REVIEW.md` | 当天小修改逻辑清单与审查发现 |

同一天的大模块序号从 `001` 递增。Code Review 批次从 `01` 递增。ID 创建后永不复用，即使文档被取消或替代。

## 长期事实与历史档案

- `docs/`：描述项目现在是什么，内容应保持当前有效。
- `MODULE_INDEX.md`：描述模块现在由谁负责、边界在哪里、从哪里开始阅读。
- `AIwork/`：描述某次重大改变为什么发生，以及某日审查了哪些小修改。

大模块关闭前，必须把仍然有效的产品、架构、接口或安全结论同步到 `docs/`，再把模块入口和文档链接同步到 `MODULE_INDEX.md`。

## 大模块生命周期

```text
Draft → Approved → In Progress → Implemented → Verified → Closed
```

| 状态 | 进入条件 |
|---|---|
| `Draft` | 已完成需求判级并建立主文档 |
| `Approved` | 用户明确批准范围，文档记录批准人和日期 |
| `In Progress` | 已开始执行批准后的实施步骤 |
| `Implemented` | 实现完成，尚待完成全部验收 |
| `Verified` | 验收标准和测试方案已执行并记录结果 |
| `Closed` | 长期文档、模块索引、工作记录和提交记录均已同步 |

只有用户可以批准大模块。Agent 可以在验证通过并完成同步后关闭文档；如果验收标准明确要求用户验收，则必须等待用户确认后关闭。

### 提交边界

1. 设计提交：只包含已批准的主文档及必要的“规划中”索引变化。
2. 实现提交：包含批准范围内的实现和当天工作记录。
3. 关闭提交：包含最终状态、验收结果、长期文档和模块索引同步。

实现可以分成多个提交，但不得把首次设计批准和完整实现折叠为同一提交。

## 跨天规则

- 主文档永远保留在创建日期目录，是该模块工作的唯一事实源。
- 后续日期为同一模块创建一个 `MOD-原始ID-WORKLOG.md`。
- 同一模块同一天只有一个工作记录；多次工作按时间追加。
- 工作记录必须链接主文档，不重新复制完整需求或设计。
- 设计发生偏差时，先在工作记录说明，再更新未关闭的主文档决策记录。

## Code Review 规则

Code Review 只在用户主动发起时创建或更新，不要求每次小修改后立即写文档。

### 审查范围

1. 查找最近日期的 `CODE_REVIEW.md`，读取最后一个已完成批次的 `head SHA`。
2. 审查从该 SHA 到当前 `HEAD` 的提交，同时审查暂存区和未提交工作区。
3. 没有历史 Review 时，以最近一个已关闭大模块的实现基线为起点；仍不存在时，以治理体系建立后的提交为起点。
4. 文档、格式和纯配置差异可以辅助理解，但不写入“逻辑修改清单”。

### 记录范围

记录：

- 单模块错误修复、内部重构、交互逻辑和错误处理。
- 每项修改的行为前后差异、原因、风险、关键路径和验证。
- Review 发现、严重级别、文件位置、处理结果和遗留项。

不记录：

- 纯构建配置、格式化、编辑器设置、锁文件和环境变量值。
- 已由大模块主文档完整覆盖且没有偏差的实现内容。

配置一旦影响运行逻辑、鉴权、安全、接口、持久化或兼容性，就必须按逻辑变化处理并重新判级。

### 基线推进

- 只有已经提交且完成审查的逻辑变化才能写入新的 `head SHA`。
- 未提交变化标记为 `Pending`，不得作为下一次 Review 的起点。
- Review 文档提交本身不属于逻辑变化；下一次 Review 可以从上次记录的代码 `head SHA` 开始，并忽略纯文档提交。

## 模板使用

- 大模块使用 [`MODULE_SPEC.md`](_templates/MODULE_SPEC.md)。
- 跨日记录使用 [`MODULE_WORKLOG.md`](_templates/MODULE_WORKLOG.md)。
- 每日审查使用 [`CODE_REVIEW.md`](_templates/CODE_REVIEW.md)。
- 模板中的必填字段不得删除；无内容时填写 `None` 或明确说明原因。

## 关闭后修订

`Closed` 主文档只能追加如下格式的勘误：

```markdown
## 勘误 YYYY-MM-DD

- 原记录：
- 修正内容：
- 修正原因：
- 影响范围：
```

如果修正会改变设计或模块行为，必须新建大模块文档，并通过 `supersedes` 指向旧文档。
