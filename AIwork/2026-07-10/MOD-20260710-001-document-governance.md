# 项目严格文档治理体系

## 目标

- 建立长期文档、按日工作档案和模块导航三层文档体系。
- 要求 Agent 在任何开发工作前判断需求属于大模块还是小修改。
- 让大模块严格执行“文档、批准、实施、验证、关闭”的流程。
- 让小修改保持快速开发，并在用户发起 Code Review 时集中归档逻辑变化。
- 让新 Agent 无需扫描全部代码即可理解模块职责、边界、依赖和入口。

## 非目标

- 不生成产品业务代码。
- 不选择前端技术栈或 AI 服务商。
- 不为纯构建配置、格式化、锁文件或环境变量调整建立逻辑变更记录。
- 不用每日档案替代 `docs/` 中长期有效的产品和架构事实。

## 判级理由

本变更会新增仓库级工作规范，约束所有后续 Agent 和全部模块的开发、审查及归档方式，属于跨模块且影响长期维护方式的大模块。

## 影响模块

- 新增项目治理模块 `GOVERNANCE`。
- 为所有规划模块建立统一索引格式和读取入口。
- 不改变任何产品模块的运行时行为。

## 当前行为

- `docs/` 已包含产品目标、架构、AI API 约定和路线图。
- 各源码规划目录只有职责说明。
- 尚无根目录 Agent 工作规范、模块索引、日期档案或 Code Review 模板。
- 后续 Agent 需要自行判断应该阅读哪些文档和代码。

## 目标行为

- 根目录 `AGENTS.md` 强制执行变更分级和最小上下文读取流程。
- 根目录 `MODULE_INDEX.md` 成为模块职责与依赖的唯一导航索引。
- `AIwork/` 按香港时区日期保存大模块主文档、跨日工作记录和每日 Code Review。
- 大模块只有在用户明确批准后才能实施；小修改无需事前文档。
- 关闭大模块时同步长期文档和模块索引，历史档案只追加勘误，不重写结论。

## 文档接口与数据流

### 固定命名

- 大模块 ID：`MOD-YYYYMMDD-NNN`。
- Code Review ID：`CR-YYYYMMDD-NN`。
- 日期目录：`AIwork/YYYY-MM-DD/`。
- 主文档：`MOD-YYYYMMDD-NNN-<ascii-slug>.md`。
- 跨日记录：`MOD-YYYYMMDD-NNN-WORKLOG.md`。
- 每日审查：`CODE_REVIEW.md`，同日多次审查按 Review ID 追加。

### 状态流

```text
Draft → Approved → In Progress → Implemented → Verified → Closed
```

- `Approved` 必须记录批准人和批准日期。
- `In Progress` 表示已开始执行批准范围。
- `Implemented` 表示文件或代码已完成，但尚未完成全部验证。
- `Verified` 表示验收项已通过。
- `Closed` 表示长期文档、模块索引和提交记录已同步。

### 信息流

```text
用户需求
  → Agent 变更判级
  → 大模块：AIwork 主文档 → 用户批准 → 实施/工作记录 → docs/MODULE_INDEX
  → 小修改：直接实施 → 用户发起 Code Review → 当日 CODE_REVIEW
```

## 失败场景

- 边界不清却被误判为小修改：默认升级为大模块。
- 大模块未批准即开始实现：必须停止实现并恢复到文档阶段。
- 模块索引与代码不一致：当前任务必须包含索引修正。
- 跨天复制完整主文档造成多个事实源：只保留创建日主文档，后续日期使用工作记录。
- 未提交修改被当作已审基线：未提交内容可以审查，但不得推进 Review 的 `head SHA`。
- 纯配置变更污染逻辑记录：从 Code Review 逻辑清单中排除；若影响运行逻辑或安全则重新判级。

## 安全和性能影响

- 本变更不处理凭据，也不改变运行时安全边界。
- 治理规则要求任何鉴权、安全、计费、并发和数据迁移变化自动按大模块处理。
- 文档索引用于减少新 Agent 的无关代码扫描，降低上下文消耗。

## 实施步骤

1. 提交本批准文档，建立设计基线。
2. 新增根目录 `AGENTS.md` 和 `MODULE_INDEX.md`。
3. 新增 `AIwork/README.md` 与三个固定模板。
4. 在模块索引中登记治理模块和八个规划产品模块。
5. 更新根 README，使治理入口可发现。
6. 检查命名、必填字段、链接、模块 ID 和工作流程示例。
7. 更新本文档状态、验证结果和提交记录，独立提交关闭材料。

## 验收标准

- 新 Agent 能从 `AGENTS.md` 找到模块索引和治理规则。
- 模块索引包含稳定 ID、职责、非职责、入口、数据、依赖、不变量、失败模式和文档链接。
- 大模块、跨日记录和 Code Review 模板字段完整。
- 治理规则能明确区分新增 AI 服务商、单模块错误修复和纯配置变更。
- 所有本地 Markdown 链接有效，模块 ID 无重复。
- Git 历史保留设计、实施和关闭三个独立阶段。

## 测试方案

- 检查所有新增文件的路径和命名格式。
- 搜索模块索引中的重复 ID。
- 解析并验证 Markdown 相对链接指向存在的文件。
- 模拟大模块、小修改、纯配置和跨天开发四类流程。
- 检查工作区、提交历史和远端 `master` 状态。
